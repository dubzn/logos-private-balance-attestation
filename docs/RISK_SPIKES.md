# Risk Spikes

The project should be built modularly. Each slice must prove one assumption
before the next layer depends on it.

## Blocker 0: On-Chain Proof Path

LP-0005 requires a LEZ verifier program that accepts/verifies a proof and gates
an on-chain action. This is the highest-risk item. If it fails, the rest of the
project may still be educational, but it is not a viable prize submission.

### Spike 0A: Direct Receipt Verification

Question:

```text
Can a LEZ guest verify a RISC Zero receipt generated by another method?
```

Steps:

1. Create a tiny RISC Zero guest that proves `x + 1 = y`.
2. Generate a real receipt with RISC Zero `3.0.5`.
3. Create a minimal LEZ guest program that receives or embeds the receipt,
   expected image id, and expected journal.
4. Submit a local sequencer transaction that runs the verifier guest.
5. Confirm a valid receipt gates one public account update.
6. Confirm a tampered receipt or journal fails with a deterministic error.

Pass condition:

```text
Valid toy receipt accepted by LEZ verifier program on local sequencer.
```

### Spike 0B: Recursive Or Native Verifier Path

Question:

```text
If direct receipt verification is too heavy or unavailable, does LEZ expose
another supported verifier path?
```

Steps:

1. Inspect current LEZ/RISC Zero verifier support.
2. Test recursive/composed proof verification if available.
3. Test any LEZ-native verifier API if exposed.
4. Document build/runtime limits.

Pass condition:

```text
At least one supported verifier path works in a minimal LEZ program.
```

### Spike 0C: Logos-Native Private Execution Gate

Question:

```text
Can the on-chain path be implemented as a native LEZ private execution where
the LEZ protocol verifies the private execution proof?
```

Why this matters:

LEZ private execution already runs a program over private account state and the
sequencer verifies the resulting proof. A balance-gate program could read a
private account, check `balance >= threshold`, and write a public gate claim
without nesting one RISC Zero receipt inside another.

This may be the most Logos-native design, but it must be confirmed with prize
evaluators because LP-0005 asks for a proof that can be submitted to a verifier
program.

Pass condition:

```text
Evaluator confirms this satisfies the on-chain path, or we keep it only as a
development fallback.
```

## Modular Build Order

After Blocker 0 passes, build in this order:

1. `attestation-core`: pure types, hashing, context ids, nullifiers, errors.
2. `lez-commitment-adapter`: exact compatibility with LEZ commitment code.
3. `sequencer-adapter`: real `getProofForCommitment`.
4. `methods/guest`: balance threshold circuit with local tests.
5. `attestation-prover`: witness builder plus RISC Zero prover.
6. `attestation-verifier`: off-chain receipt verification.
7. `attestation-cli`: thin commands around the core pieces.
8. `lez/verifier-program`: selected on-chain path.
9. Messaging adapter and token-gated demo.
10. Basecamp GUI.
11. External integrations and submission hardening.

Every item should have a local test or demo command before the next item starts.

## Stop Conditions

Pause and re-plan if:

- no on-chain proof path works
- RISC Zero version compatibility breaks against LEZ
- the sequencer cannot return membership proofs for initialized private accounts
- presenter binding is rejected as insufficient by evaluators
- Messaging requirements cannot be met with available Logos tooling
